# Исследование панели продавца BakeBay

## Резюме

Панель продавца BakeBay — это полнофункциональный инструмент управления для индивидуальных кондитеров. Проект построен
на Next.js 15 (App Router), использует PostgreSQL через Drizzle ORM, UI на Radix + Tailwind, графики на Recharts. Панель
включает 7 разделов: Обзор, Товары, Заказы, Ингредиенты, Отзывы, Вопросы и Настройки (не реализован).

Ниже — детальный разбор каждого аспекта с позиции индивидуального кондитера, которому нужно вести своё дело.

---

## 1. Сильные стороны

### 1.1. Раздел «Ингредиенты» — ключевая ценность продукта

Это самый сильный и самый уникальный раздел панели. Для индивидуального кондитера управление ингредиентами — ежедневная
боль, и здесь она решена на нескольких уровнях:

- **Список покупок** автоматически агрегирует все ингредиенты из «ожидающих» заказов. Кондитер открывает вкладку и
  видит, что нужно купить прямо сейчас. Чекбоксы с прогресс-баром (процент выполненных закупок) — отличный UX-паттерн,
  создающий ощущение контроля.
- **Группировка по заказам** позволяет понять, какие ингредиенты для какого заказа нужны. Это критически важно при
  работе с несколькими заказами одновременно.
- **Инвентарь ингредиентов** с inline-редактированием (stock, alert threshold, цена закупки) — позволяет быстро
  обновлять данные без переходов по страницам.
- **LP-оптимизация** — продвинутая функция с двумя режимами (минимизация закупок и максимизация прибыли). Для
  индивидуального кондитера это уникальная ценность, которую сложно найти в аналогах.

### 1.2. Канбан-доска заказов

Переключение между списком и канбан-доской — сильное решение. Канбан с колонками (Новые → В обработке → Отправлены →
Доставлены) визуально показывает весь поток заказов. Кнопки быстрого перевода статуса (`→ В обработке`) прямо на
карточке канбана позволяют управлять заказом одним кликом.

### 1.3. Экспорт данных

Возможность экспорта в CSV реализована в нескольких местах: смета заказа, смета продукта, инвентарь закупок. Для
индивидуального кондитера, который ведёт финучёт в Excel или передаёт данные бухгалтеру, это практичная функция.

### 1.4. Система уведомлений

Поллинг новых заказов каждые 30 секунд и новых отзывов каждые 60 секунд — хорошее решение для MVP. Уведомления об
ингредиентах с низким запасом (`useIngredientAlerts`) помогают не пропустить момент, когда нужно докупить продукты.

### 1.5. Управление отзывами

Разделение на вкладки «Ожидают ответа» / «С ответом» — правильный паттерн, помогающий кондитеру не пропускать отзывы без
ответа. Для малого бизнеса ответы на отзывы — критически важны для репутации.

### 1.6. Обзорная страница (дашборд)

Четыре KPI-карточки (доход, заказы, товары, клиенты/рейтинг) с индикаторами изменения (% к прошлому месяцу) дают быстрый
snapshot бизнеса. График дохода за 12 месяцев, последние заказы и отзывы, топ товаров — всё собрано компактно.

### 1.7. Русская локализация

Весь интерфейс на русском языке — от навигации до статусов и дат (date-fns с локалью `ru`). Для целевой аудитории (
российские кондитеры) это обязательное требование, и оно выполнено.

## 2. Слабые стороны и проблемы

### 2.1. Страница «Настройки» не реализована

В навигации есть ссылка на `/seller-dashboard/settings`, но самой страницы нет. Это критический пробел: кондитеру негде
настроить информацию о себе, свой магазин, способы связи, график работы, зоны доставки, реквизиты и прочее.

### 2.2. Страница «Вопросы» работает на моковых данных

Раздел «Вопросы» не подключён к реальному API — используются захардкоженные примеры. Функция существует в UI, но не
работает на практике.

### 2.3. Захардкоженные категории товаров

В фильтре категорий на странице товаров список категорий зашит прямо в код: «Торты», «Печенье», «Выпечка», «Итальянские
десерты», «Шоколад», «Капкейки». Категории не подтягиваются из базы и не позволяют кондитеру создавать свои.

### 2.4. Fallback на примеры товаров

Если загрузка товаров из API не удалась, показываются захардкоженные `exampleProducts` с фиктивными данными (Шоколадный
торт за $24.99 и т.д.). Пользователь может принять фейковые данные за реальные.

### 2.5. Нет подтверждения опасных действий

Пункт «Удалить товар» в дропдаун-меню не реализован (нет обработчика `onClick`), но даже если он появится — нет
подтверждающего диалога. Аналогично «Обновить запасы» — нет обработчика.

### 2.6. Статус оплаты всегда «Оплачен»

В карточке заказа поле «Статус оплаты» всегда показывает `<Badge>Оплачен</Badge>` — это захардкожено. Реальный статус
оплаты из данных не используется.

### 2.7. Смена статуса заказа без ограничений

В списке заказов продавец может перевести заказ в любой статус из любого текущего. Дропдаун содержит ВСЕ статусы (
включая «Новый» и «Отменён»), хотя в канбан-доске реализован корректный flow (ordering → processing → in_progress →
delivering → delivered). Нет защиты от перевода заказа назад или в некорректный статус.

### 2.8. Нет пагинации

Ни заказы, ни товары, ни отзывы не имеют пагинации. Все данные загружаются разом. При росте бизнеса (100+ заказов, 50+
товаров) страницы начнут тормозить.

### 2.9. Ошибки молча проглатываются

В нескольких местах ошибки обрабатываются через `catch { // silently fail }` (например, загрузка отзывов). Пользователь
не видит, что что-то пошло не так.

### 2.10. Ссылка «Центр поддержки» ведёт на личный Telegram

В навигации блок «Нужна помощь?» ссылается на `https://t.me/tarasov_vasili1` — это личный Telegram-аккаунт, а не центр
поддержки. Текст «Перейти в Центр поддержки» вводит в заблуждение.

## 3. Чего не хватает

### 3.1. Календарь заказов / планирование

Для кондитера ключевой инструмент — понимание загрузки по дням. «Сколько заказов на вторник? Успею ли?» Нет ни
календаря, ни timeline-вида заказов с привязкой к дате выполнения/доставки. Это одна из самых востребованных функций в
сфере custom-выпечки.

### 3.2. Общение с клиентом

Нет встроенного чата или системы сообщений между продавцом и покупателем. Кондитеру часто нужно уточнить детали заказа (
надпись на торте, аллергии, время доставки). Сейчас единственный канал — Q&A, который работает на моках.

### 3.3. Финансовая аналитика

Есть «Общий доход» и «Ср. чек», но нет:

- Расчёта себестоимости (cost) vs. выручки (revenue) = маржинальность
- Учёта расходов на ингредиенты (данные есть в инвентаре, но не сводятся в финотчёт)
- P&L отчёта даже в простой форме
- Прогноза доходов

### 3.4. Управление рецептами / шаблоны продуктов

Привязка ингредиентов к продуктам есть, но нет полноценного управления рецептурами. Кондитеру нужно: задать рецепт
торта → автоматически считать себестоимость → масштабировать на разные размеры. Сейчас ингредиенты привязаны к продукту,
но без понятия «рецепт».

### 3.5. Фото-галерея работ / портфолио

Для кондитера визуальное портфолио — главный маркетинговый инструмент. В панели нет раздела для загрузки фотографий
выполненных работ, которые показываются на витрине продавца.

### 3.6. Мобильная адаптация навигации

Layout использует `flex-col md:flex-row` для разделения навигации и контента, но боковая навигация не имеет мобильного
меню (hamburger/drawer). На мобильных все ссылки показываются списком сверху, занимая экран.

### 3.7. Печать заказов

Нет кнопки «Печать заказа» для вывода на принтер (наклейка, накладная). Многие кондитеры распечатывают заказы и
прикрепляют к коробке.

### 3.8. Bulk-операции

Нет массового изменения статуса заказов, массового обновления цен или инвентаря товаров. При работе с десятками позиций
это критично.

### 3.9. Уведомления в приложении

Уведомления реализованы через polling + toast, но нет панели/журнала уведомлений. Если кондитер пропустил toast, он не
узнает о новом заказе до тех пор, пока не зайдёт в раздел «Заказы».

## 4. Что может быть лишним или запутывающим

### 4.1. LP-оптимизация для начинающих

Вкладка «LP-оптимизация» с терминами «минимум закупок» и «максимум прибыли» — мощный инструмент, но может запутать
целевую аудиторию. Начинающий кондитер не знает, что такое LP. Нужны подсказки, туториал или более понятное название (
например, «Помощник по закупкам»).

### 4.2. Дублирование фильтрации заказов

На странице заказов в режиме списка есть одновременно: табы (Все/В обработке/Отправлены/Доставлены) и дропдаун с
фильтром по статусу. Они работают параллельно и могут конфликтовать, запутывая пользователя.

### 4.3. Вкладка «Топ товаров» на дашборде

На главной странице внизу уже есть блок «Топ товаров» внутри вкладки «Обзор». Вторая вкладка «Топ товаров» дублирует эту
информацию с чуть более детальной таблицей. Для кондитера с 5-10 товарами это избыточно.

## 5. Анализ по разделам: насколько удобно для ведения дела

### Обзор (Dashboard) — 7/10

Хорошая сводка, но не хватает «что делать прямо сейчас» — сколько заказов требуют внимания, какие ингредиенты
заканчиваются, есть ли неотвеченные отзывы. Сейчас это просто цифры, без призыва к действию.

### Товары — 6/10

Базовый CRUD есть, но управление каталогом для кондитера недостаточно: нет вариантов товара (один торт разных
размеров/начинок), нет шаблонов, нет привязки к конкретным рецептам с автоматическим расчётом себестоимости.

### Заказы — 7/10

Канбан — сильная сторона. Но: нет даты выполнения/дедлайна заказа, нет комментариев к заказу от продавца, нет печати,
нет привязки к календарю.

### Ингредиенты — 9/10

Лучший раздел. Список покупок, группировка по заказам, инвентарь с inline-редактированием, оптимизация. Для доработки:
добавить историю цен, автозаказ у поставщиков, привязку к конкретным магазинам.

### Отзывы — 7/10

Функциональный раздел с правильными фильтрами. Не хватает: шаблонов ответов (кондитеры часто пишут похожие
благодарности), агрегированной статистики по рейтингу (средний за месяц, динамика).

### Вопросы — 2/10

Не работает (мок-данные). Потенциально полезный раздел, но в текущем виде не несёт ценности.

### Настройки — 0/10

Не реализован вообще.

## 6. Рекомендации по приоритетам доработки

**Критические (влияют на работоспособность):**

1. Реализовать страницу Настройки (профиль магазина, контакты, зоны доставки)
2. Подключить Вопросы к реальному API
3. Убрать fallback на фейковые данные в товарах
4. Исправить захардкоженный статус оплаты

**Высокий приоритет (повышают ценность для кондитера):**

5. Добавить календарь заказов
6. Реализовать систему сообщений продавец-покупатель
7. Добавить пагинацию
8. Ограничить переходы между статусами заказа в режиме списка

**Средний приоритет (улучшают UX):**

9. Мобильная навигация (drawer)
10. Action-items на дашборде («3 заказа ждут обработки», «2 ингредиента на исходе»)
11. Шаблоны ответов на отзывы
12. Печать заказа
13. Динамические категории товаров

**Низкий приоритет (nice-to-have):**

14. Финансовая аналитика (P&L)
15. Портфолио работ
16. Управление рецептурами
17. WebSocket вместо polling
